# 專案概覽 (Project Overview)

本目錄 (`d:\請假單關卡`) 包含公司請假單系統的工作流引擎審核邏輯（關卡判定）。專案主要由多個 Python 腳本組成，每個腳本代表一個特定的簽核關卡或職位（如：`500單位主官.py`、`1000工務副總.py` 等）。

這些腳本並非獨立運行的應用程式，而是作為「業務規則引擎」的判斷邏輯。主系統會在執行時，將請假單的相關屬性（如請假時數、假別、申請人職級等）作為全局變數注入腳本中，並根據腳本回傳的 `True`（需經過此關卡審核）或 `False`（跳過此關卡）來決定流程走向。

## 目錄結構與關鍵檔案 (Directory Structure & Key Files)

*   **`參數說明.md`**: 核心數據字典。定義了腳本中所有變數的意義與代碼對應關係，包含：
    *   數值與狀態旗標（如：`hours` 請假時數, `continueDays` 連續天數）。
    *   代碼變數的詳細映射表（如：`plevel` 簽核部門, `nlevel` 職級, `vaid` 假別名稱 等）。**在修改或閱讀腳本時，必須對照此文件。**
*   **`[數字][職稱].py`**: 各關卡的判斷腳本。檔名前綴的數字通常對應職級或關卡順序（如 `400`, `500`, `1000` 等）。腳本內容主要是基於 `if-else` 的條件判斷，最終 `return True` 或 `return False`。

## 開發與維護慣例 (Development & Maintenance Conventions)

1.  **無狀態判斷 (Stateless Logic):** 腳本僅能依賴系統注入的變數進行判斷，不能包含複雜的物件導向設計、外部 API 呼叫或資料庫連線。腳本結構應保持簡單，由一系列的 `if` 條件和 `return` 組成。
2.  **變數參照:** 修改邏輯時，必須嚴格依循 `參數說明.md` 中的變數定義。例如，判斷是否為事假必須使用 `vaid == 12`。
3.  **註解文化:** 由於業務邏輯繁雜（包含多種特定假別、部門特殊規定），在新增或修改條件時，應加入清晰的中文註解，說明該條件對應的實際行政規定或修改者（如代碼中常見的 `#彬鴻修改`），以便後續追蹤。
4.  **防禦性回傳 (Default Return):** 每個腳本的結尾通常會有一個預設的回傳值（通常是 `return False` 或 `return True`，視該關卡的預設行為而定），以防所有特殊條件都未命中。
5.  **測試與驗證:** 由於這些是系統動態執行的規則片段，修改後應透過主系統的測試環境，模擬對應的請假單參數（`hours`, `vaid`, `nlevel` 等），確認流程是否如預期般進入或跳過該關卡。
