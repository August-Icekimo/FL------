# 請假單關卡判定腳本集

## 概述

本目錄下每支 `.py` 腳本對應一個簽核關卡，由主系統動態執行，判斷請假單是否需要經過該關卡。腳本不是獨立程式，主系統在執行時將請假單屬性注入為全域變數，腳本邏輯讀取這些變數並 `return True`（需審核）或 `return False`（跳過）。

檔案命名規則：`[數字][職稱].py` —— 數字代表關卡職等順序（數字越大職位越高）。

---

## 邏輯流程設計原則

### 三階段執行順序

1. **快速全局排除**（第17-24行）
   - 規則1：假別本身在快速排除名單 → 直接排除
   - 規則2：部門/職級排除（但防疫會務假例外） → 直接排除

2. **強制進入**（第28-35行）
   - 防疫會務假強制進入
   - 副主管以下 + 總廠人員的長天數/特殊假別強制進入

3. **排除條件**（第39-45行）
   - 副主管特定短天數/假別排除
   - 非一二廠副主管以下的補休及公出排除

## 關鍵設計決策與說明

### 1. 假別15（特准病假）的邊界邏輯 ✓

**現象**：假別15在總廠強制進入，在非總廠被排除

**原因**：假別15須重大傷病卡才能額外核准，通常是**補登後簽**（非自主申請），補登只出現在總廠路徑，故邏輯正確。

**涉及行**：第33-35行、第40行

---

### 2. 假別21（遞延休假）被快速排除 ✓

**現象**：假別21在 `vaid_quick_exclude` 中，被第一階段直接排除

**原因**：遞延休假視作**積欠未休假**，審核上較寬鬆直接放過，不需進入人事副主任簽核。

**涉及行**：第19-20行

**備註**：假別21同時在 `vaid_common` 和 `vaid_750_short_term` 中，但不會因此進入第33-35行或第40行，因為規則1優先執行。

---

### 3. nlevel >= 800（職級主管以上）排除 ✓

**現象**：職級 >= 800（人事室主任/副主任級）的假別直接排除

**原因**：該職級已處於同儕層級，不做同儕批准（peer review）。應由更高層級或平行部門簽核，跳過人事副主任。

**涉及行**：第23行

**例外**：防疫會務假在規則2中已排除，但第29-30行會再次強制進入（優先級更高）。

---

### 4. plevel=6（業務企劃中心）排除 ✓

**現象**：業務企劃中心人員的假別無法進入第33-35行邏輯

**原因**：該部門有獨立簽核路徑，不納入人事副主任的一般簽核範疇。

**涉及行**：第33行條件 `plevel in (3, 4)` 不包含6

---

### 5. vaid_exclude_with_makeups 補休排除 ✓

**現象**：補修類假別（35-38, 109-111）在特定條件下被排除

**原因**：補休屬於內部結算性質，無需人事副主任審批，直接排除是正確的。

**涉及行**：第40行、第44行

---

## 已驗證的「灰色地帶」

### ✅ 假別7、21的雙重身份風險 — 風險可接受

假別7（公假）和21（遞延休假）同時在 `vaid_quick_exclude` 和 `vaid_common` 中，導致某些情況下無法進入第33-35行的長期假邏輯。

**結論**：風險可接受，因為：
- 假別7和21都是應該快速排除的假別
- 需要特殊審核的情況（高職級、長期假）由其他關卡負責，不是人事副主任的責任

### ✅ 規則1粒度過粗風險 — 風險可接受

規則1無視職級、天數，直接排除整個假別類別。

**結論**：風險可接受，因為：
- 假別5、6都是特殊假別，應該快速排除
- 假別7（公假）和21（遞延休假）由前述說明，排除是正確的

## 維護注意事項

1. **假別常數變更**：修改 `vaid_quick_exclude`、`vaid_common`、`vaid_meeting_pandemic` 時，需檢查是否會影響強制進入邏輯。特別是假別7、21等多重身份假別。

2. **職級邊界**：nlevel的邊界在750（副主管）和800（主管級）。修改時應確認第33-35行、第39-44行的判斷是否仍然連貫。

3. **部門特殊性**：plevel=1,2有獨立排除邏輯；plevel=3,4有特殊進入邏輯；plevel=6有獨立路徑。修改時需確認是否遺漏其他部門代碼。

4. **補休補登邏輯**：假別15、補休類假別的邏輯與補登流程緊密相關，修改前應確認與主系統補登機制的一致性。
