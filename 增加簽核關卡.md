# 📌 作業目的與情境

當請假單某些簽核關卡**無故缺漏或未被觸發**時，應優先從根本排除**流程判定條件問題**，再透過「重新送單 + 複製簽核紀錄」方式恢復完整的審核路徑。

**典型場景**：
- 某關卡（如 800=單位主官、700=組長）因判定腳本誤差，fst_isValid = 0（條件未進入）
- 請假單在中途某個職級被卡住，導致高層審批（如 900=副總、1100=總經理）無法進入
- 需要補齊缺漏的簽核審批人員

**優先順序**：
1. **診斷** — 檢查關卡定義與判定條件是否正確
2. **修復** — 調整流程判定腳本，使關卡能正確進入
3. **恢復** — 重新送單並複製先前的簽核紀錄到新申請單（推薦）
4. **備選** — 直接在資料庫修改現有申請單的流程狀態（謹慎操作）

# ⚠️ 核心限制與安全防禦

1. **唯一鍵衝突防範**：flow_state_uid 表以 (fsu_did, fsu_serial, fsu_uid) 為複合唯一鍵，新增記錄前須確保不重複。
2. **關卡序號遞增原則**：fsu_serial 必須遞增，新增關卡應使用比現有最大 serial 更大的值。
3. **instord 連續性檢查**：新增關卡的 instord（職級順序）須符合簽核順序邏輯，勿破壞既有流程。
4. **主檔同步**：修改 flow_instance 的 flow_instord 指標，確保前台顯示與資料庫狀態一致。
5. **操作隔離**：所有異動應包裹在 Transaction 中，執行完畢確認無誤後才 COMMIT。

## 🔍 第一階段：診斷問題（Diagnose & Validate）

### 步驟一：檢查缺漏關卡的狀態

**企圖 (Intent)**：診斷為何關卡未進入簽核流程。關卡缺漏的主要原因是 `fst_isValid = 0`（條件判定未進入），或 `flow_state_uid` 表中沒有對應的審批人員記錄。

**操作步驟 (Steps)**：
1. 修改 @TargetFlowId 為目標請假單的 flow_id。
2. 執行查詢，檢查需要進入的關卡（如 700=組長、800=單位主官）的 fst_isValid 狀態。
3. 若 fst_isValid = 0，說明判定條件未觸發，需修改流程判定腳本；若 fst_isValid = 1 但 flow_state_uid 無人員，則需補齊審批者。

```sql
-- ==========================================
-- 步驟一：診斷缺漏關卡的根本原因
-- ==========================================
DECLARE @TargetFlowId INT = 649353; -- 🔍 目標請假單的 flow_id

-- 【診斷 A】檢查 flow_state 表的關卡定義與啟用狀態
SELECT 
    fst_serial AS [流水號],
    fst_instord AS [職級順序],
    fst_name AS [關卡名稱],
    fst_isValid AS [是否啟用_0未進入_1已進入],
    fst_isClose AS [是否結束],
    fst_stime AS [進入時間]
FROM edoc..flow_state
WHERE fst_did = CAST(@TargetFlowId AS VARCHAR(50))
ORDER BY fst_instord ASC;

-- 【診斷 B】檢查 flow_state_uid 表的審批人員（若為空，表示雖定義了但未派員）
SELECT 
    fsu_serial AS [流水號],
    fsu_instord AS [職級順序],
    fsu_name AS [簽核人名],
    fsu_status AS [簽核狀態_0待簽_1已簽],
    COUNT(*) OVER (PARTITION BY fsu_instord) AS [該關卡人數]
FROM edoc..flow_state_uid
WHERE fsu_did = @TargetFlowId
ORDER BY fsu_instord ASC;

-- 【診斷 C】對比缺漏：應進入但未進入的關卡清單
SELECT 
    'flow_state 有定義' AS [狀態],
    s.fst_instord,
    s.fst_name,
    s.fst_isValid
FROM edoc..flow_state s
WHERE s.fst_did = CAST(@TargetFlowId AS VARCHAR(50))
  AND NOT EXISTS (
    SELECT 1 FROM edoc..flow_state_uid u
    WHERE u.fsu_did = @TargetFlowId
      AND u.fsu_instord = s.fst_instord
  )
ORDER BY s.fst_instord ASC;
```

### 步驟二：追溯問題來源 — 檢查流程判定腳本

**企圖 (Intent)**：若關卡的 fst_isValid = 0，表示該關卡的判定腳本條件未觸發。需檢查對應的判定腳本（如 `700組長.py`、`800單位主官.py`）是否邏輯正確。

**操作步驟 (Steps)**：
1. 根據步驟一診斷出的缺漏職級，對應查找判定腳本（參考 CLAUDE.md 中的檔案命名規則）。
2. 檢查該腳本的判定條件（基於 vaid、nlevel、plevel 等全域變數）。
3. 若判定條件有誤，修改並重新測試。
4. **重點**：不要直接修改資料庫 flow_state 的 fst_isValid，應從根本修正判定腳本，讓流程能自動觸發。

**相關檔案查詢**：

```bash
# 查找該職級的判定腳本（例如查找 700 相關的腳本）
ls 請假單/ | grep "700"
```

**常見判定問題**：
- 補假別判定誤差（需檢查 vaid 黑名單/白名單）
- 部門層級判定誤差（需檢查 plevel、second 等部門識別碼）
- 職級判定誤差（需檢查 nlevel）

### 步驟三：查找相同情境的完整申請單（作為複製來源）

**企圖 (Intent)**：若要恢復流程，最安全的做法是找到一份**條件相同、流程完整**的歷史申請單，將其簽核紀錄複製到目標單據。這樣可避免直接在資料庫新增簽核記錄的風險。

**操作步驟 (Steps)**：
1. 搜尋歷史申請單，找一份**相同假別、相同申請者職級、流程完整**的單據。
2. 記錄該完整申請單的 flow_id（稱為 @SourceFlowId）。
3. 確認其 flow_state_uid 表包含所需的職級層級（如 700、800、900 等）。

```sql
-- ==========================================
-- 步驟三：查找相同條件的完整申請單（複製來源）
-- ==========================================

-- 假設目標申請單的條件（請根據實際調整）
DECLARE @TargetFlowId INT       = 649353;   -- 🔍 有缺漏的申請單
DECLARE @TargetUid INT          = 1302;     -- 🔍 申請人 uid
DECLARE @SearchDays INT         = 30;       -- 🔍 向前查詢天數

-- 查找流程完整的歷史申請單（條件相同的參考單據）
SELECT TOP 20
    fi.flow_id,
    fi.flow_uid,
    fi.flow_createtime AS [送單時間],
    (SELECT COUNT(DISTINCT fsu_instord) 
     FROM edoc..flow_state_uid 
     WHERE fsu_did = fi.flow_id) AS [簽核職級數],
    fi.bfm_leaveName AS [假別]
FROM edoc..flow_instance fi
WHERE fi.flow_uid = @TargetUid
  AND fi.flow_patternid = '60'  -- 請假單
  AND fi.flow_closeType IN (0, 2)
  AND fi.flow_createtime >= DATEADD(DAY, -@SearchDays, GETDATE())
  AND fi.flow_id != @TargetFlowId
ORDER BY (SELECT COUNT(*) FROM edoc..flow_state_uid WHERE fsu_did = fi.flow_id) DESC;

-- 驗證該申請單的流程是否完整（查看其 flow_state_uid 人員數）
DECLARE @SourceFlowId INT = 650000; -- ⚙️ 用來複製的完整申請單 flow_id

SELECT 
    fsu_serial,
    fsu_instord,
    fsu_name,
    COUNT(*) OVER (PARTITION BY fsu_instord) AS [該職級人數]
FROM edoc..flow_state_uid
WHERE fsu_did = @SourceFlowId
ORDER BY fsu_instord ASC;
```

## 🛠️ 第二階段：修復與恢復（Fix & Recover）

### 步驟四：修復流程判定腳本（推薦做法）

**企圖 (Intent)**：根據步驟二的診斷，修改判定腳本使其正確觸發，這是最根本的解決方案。

**操作步驟 (Steps)**：
1. 定位並開啟對應的判定腳本（如 `700組長.py`、`800單位主官.py`）。
2. 檢查其中的條件判定邏輯（如 vaid 假別黑白名單、plevel 部門判定、nlevel 職級判定）。
3. 根據實際情況修改條件（例如：補假別進黑名單、調整部門範圍等）。
4. 在測試環境驗證修改後的效果。

**常見修改**：

```python
# 【範例】若發現補休假別誤被判定，需在黑名單中加入
# 原本的判定（有問題）：
if vaid == 20:  # 只判定休假
    return True

# 修改後（正確）：
if vaid in (20, 35, 36, 37, 38):  # 加入補假別
    return True
```

詳見 `請假單/README.md` 和各判定腳本內的註解。

### 步驟五：重新送單並複製簽核紀錄（推薦做法）

**企圖 (Intent)**：在判定腳本修復後，員工需要重新送一張申請單。然後透過複製先前相同條件申請單的簽核紀錄，快速恢復完整的審核流程。

**操作步驟 (Steps)**：
1. 通知申請人重新送一張新的申請單（須等流程判定腳本修復完成後）。
2. 確認新申請單的流程現在能正確進入所需的關卡。
3. 找出一份歷史申請單（相同假別、相同申請者職級、流程完整），記錄其 flow_id（@SourceFlowId）。
4. 執行複製 SQL，將來源申請單的簽核人員紀錄複製到新申請單。

```sql
-- ==========================================
-- 步驟五：複製簽核紀錄到新申請單
-- ==========================================
BEGIN TRANSACTION;

DECLARE @TargetFlowId INT      = 649353;   -- ⚙️ 重新送的新申請單 flow_id
DECLARE @SourceFlowId INT      = 650206;   -- ⚙️ 流程完整的參考申請單 flow_id

-- 【Sub-Step 1】驗證來源申請單的簽核人員（確認有完整的職級層級）
SELECT 
    fsu_serial,
    fsu_instord,
    fsu_name,
    fsu_status
FROM edoc..flow_state_uid 
WHERE fsu_did = @SourceFlowId
ORDER BY fsu_instord ASC;

-- 【Sub-Step 2】複製來源的簽核人員記錄到新申請單
--             （自動生成新的 serial 序號）
INSERT INTO edoc..flow_state_uid 
(fsu_did, fsu_serial, fsu_uid, fsu_instord, fsu_name, fsu_depid, fsu_depName, 
 fsu_status, fsu_actionid, fsu_isValid, fsu_roleClassId, fsu_roleClassName)
SELECT 
    @TargetFlowId,                           -- 新申請單 ID
    ROW_NUMBER() OVER (ORDER BY fsu_instord) + ISNULL((SELECT MAX(fsu_serial) FROM edoc..flow_state_uid WHERE fsu_did = @TargetFlowId), 0),
    fsu_uid,
    fsu_instord,
    fsu_name,
    fsu_depid,
    fsu_depName,
    '0' AS fsu_status,                       -- 重設為待簽核
    '-1' AS fsu_actionid,
    '1' AS fsu_isValid,
    fsu_roleClassId,
    fsu_roleClassName
FROM edoc..flow_state_uid 
WHERE fsu_did = @SourceFlowId;

-- 【Sub-Step 3】驗證複製結果（確認職級都進來了）
SELECT 
    fsu_serial,
    fsu_instord,
    fsu_name,
    COUNT(*) OVER (PARTITION BY fsu_instord) AS [該職級人數]
FROM edoc..flow_state_uid 
WHERE fsu_did = @TargetFlowId
ORDER BY fsu_instord ASC;

-- 【Sub-Step 4】同步更新主檔指標（如需要）
UPDATE edoc..flow_instance
SET flow_instord = (SELECT MIN(fsu_instord) FROM edoc..flow_state_uid WHERE fsu_did = @TargetFlowId)
WHERE flow_id = @TargetFlowId;

-- 確認無誤後執行 COMMIT
/*
COMMIT TRANSACTION;
-- ROLLBACK TRANSACTION;
*/
```

**優勢**：
- ✅ 新申請單的流程狀態完整且正確
- ✅ 簽核人員、部門資訊自動帶入
- ✅ 前台可正確顯示簽核歷程
- ✅ 避免直接修改舊申請單帶來的風險

### 步驟六：啟用/修改現有申請單的關卡（備選做法 — 謹慎操作）

**企圖 (Intent)**：若無法重新送單，或必須在現有申請單上補齊缺漏的關卡，需修改三個表：flow_instance、flow_state、flow_state_uid。

**前置條件**：
- flow_state 表中關卡已定義（fst_isValid = 0 代表條件未進入）
- 需要將該關卡的 fst_isValid 改為 1，並補齊 flow_state_uid 的審批人員

**操作步驟 (Steps)**：
1. 啟用 flow_state 中的目標關卡（設定 fst_isValid = 1）
2. 新增或更新 flow_state_uid 的簽核人員
3. 同步更新 flow_instance 的指標

```sql
-- ==========================================
-- 步驟六：直接修改現有申請單的關卡（高風險操作）
-- ==========================================
BEGIN TRANSACTION;

DECLARE @TargetFlowId INT      = 649353;      -- ⚙️ 要修改的申請單 flow_id
DECLARE @TargetInstOrd INT     = 800;         -- ⚙️ 要啟用的職級（如 800=單位主官）

-- 【Sub-Step 1】啟用目標職級的關卡定義
UPDATE edoc..flow_state
SET fst_isValid = '1',
    fst_stime = NULL,                        -- 清空進入時間（允許重新進入）
    fst_isClose = '0',
    fst_extminimum = '1'
WHERE fst_did = CAST(@TargetFlowId AS VARCHAR(50))
  AND fst_instord = @TargetInstOrd;

-- 【Sub-Step 2】補齊審批人員（若 flow_state_uid 無該職級記錄，則新增）
--             使用 MERGE 語法可同時處理新增與更新
DECLARE @NewSerial INT = (SELECT ISNULL(MAX(fsu_serial), 0) + 1 FROM edoc..flow_state_uid WHERE fsu_did = @TargetFlowId);
DECLARE @NewUid INT = 942;                    -- ⚙️ 簽核人員 uid
DECLARE @NewName NVARCHAR(50) = '李經理';    -- ⚙️ 簽核人名稱
DECLARE @NewDepId NVARCHAR(50) = 'HRD01';     -- ⚙️ 部門編號
DECLARE @NewDepName NVARCHAR(100) = '人事室'; -- ⚙️ 部門名稱

INSERT INTO edoc..flow_state_uid 
(fsu_did, fsu_serial, fsu_uid, fsu_instord, fsu_name, fsu_depid, fsu_depName, 
 fsu_status, fsu_actionid, fsu_isValid)
SELECT @TargetFlowId, @NewSerial, @NewUid, @TargetInstOrd, @NewName, @NewDepId, @NewDepName,
       '0', '-1', '1'
WHERE NOT EXISTS (
    SELECT 1 FROM edoc..flow_state_uid 
    WHERE fsu_did = @TargetFlowId 
      AND fsu_instord = @TargetInstOrd
      AND fsu_uid = @NewUid
);

-- 【Sub-Step 3】同步更新主檔 flow_instance 指標
UPDATE edoc..flow_instance
SET flow_instord = CAST(@TargetInstOrd AS NVARCHAR(10))
WHERE flow_id = @TargetFlowId;

-- 【Sub-Step 4】驗證修改結果
SELECT 
    '【flow_state】' AS [檢查層],
    fst_instord,
    fst_name,
    fst_isValid,
    fst_isClose
FROM edoc..flow_state
WHERE fst_did = CAST(@TargetFlowId AS VARCHAR(50))
  AND fst_instord = @TargetInstOrd
UNION ALL
SELECT 
    '【flow_state_uid】' AS [檢查層],
    fsu_instord,
    fsu_name,
    CAST(fsu_status AS NVARCHAR(10)),
    NULL
FROM edoc..flow_state_uid
WHERE fsu_did = @TargetFlowId
  AND fsu_instord = @TargetInstOrd;

-- 確認無誤後執行 COMMIT；出現錯誤（如唯一鍵衝突）則 ROLLBACK
/*
COMMIT TRANSACTION;
-- ROLLBACK TRANSACTION;
*/
```

**⚠️ 風險提示**：
- 直接修改資料庫流程可能導致前台顯示異常
- 若簽核人員已簽名，直接新增可能導致簽核順序混亂
- 建議先測試環境驗證，再在生環執行

### 步驟七：清除或重設簽核人員記錄（異常恢復）

**企圖 (Intent)**：若某簽核人員因故誤簽或需重新簽核，可清空其簽核時間、簽名等欄位，使其回到待簽核狀態。

**操作步驟 (Steps)**：
1. 精準鎖定要清除的簽核記錄（by fsu_did + fsu_serial）。
2. 執行 UPDATE 清空簽核相關欄位。
3. 驗證該人員狀態已回復為未簽核。

```sql
-- ==========================================
-- 步驟七：重設特定簽核人員的簽核記錄
-- ==========================================
BEGIN TRANSACTION;

DECLARE @TargetFlowId INT        = 649353;      -- ⚙️ 流程 ID
DECLARE @TargetSerial INT        = 4164219;     -- ⚙️ 要重設的簽核流水號

-- 清除簽核人員的簽核紀錄（回復待簽核狀態）
UPDATE edoc..flow_state_uid
SET fsu_signname = '',           -- 簽名人名稱
    fsu_ftime = NULL,            -- 簽核時間
    fsu_actionid = '-1',         -- 重設為未簽核
    fsu_status = '0'             -- 簽核狀態改為待簽核
WHERE fsu_did = @TargetFlowId
  AND fsu_serial = @TargetSerial;

-- 驗證結果
SELECT fsu_serial, fsu_instord, fsu_name, fsu_ftime, fsu_status 
FROM edoc..flow_state_uid 
WHERE fsu_did = @TargetFlowId 
  AND fsu_serial = @TargetSerial;

-- 確認無誤後執行 COMMIT
/*
COMMIT TRANSACTION;
-- ROLLBACK TRANSACTION;
*/
```

## ✅ 第三階段：驗證與異常處理（Verify & Handle Issues）

### 步驟八：操作後完整性驗證

執行以下查詢確認流程恢復正確：

```sql
-- ==========================================
-- 步驟八：操作後完整性驗證清單
-- ==========================================
DECLARE @TargetFlowId INT = 649353; -- ⚙️ 目標請假單 flow_id

-- 【驗證 1】主檔狀態 — flow_instance 指標是否指向正確關卡
SELECT 
    'flow_instance' AS [檢查表],
    flow_id,
    flow_instord AS [當前關卡指標],
    flow_closeType AS [結案狀態],
    flow_createtime AS [建立時間]
FROM edoc..flow_instance
WHERE flow_id = @TargetFlowId;

-- 【驗證 2】關卡定義層 — flow_state 是否有缺漏關卡
SELECT 
    'flow_state' AS [檢查表],
    fst_serial,
    fst_instord,
    fst_name,
    fst_isValid AS [條件是否進入],
    fst_isClose AS [是否結束]
FROM edoc..flow_state
WHERE fst_did = CAST(@TargetFlowId AS VARCHAR(50))
ORDER BY fst_instord ASC;

-- 【驗證 3】簽核人員層 — flow_state_uid 是否齊全
SELECT 
    'flow_state_uid' AS [檢查表],
    fsu_instord AS [職級],
    COUNT(*) AS [人數],
    SUM(CASE WHEN fsu_status = '0' THEN 1 ELSE 0 END) AS [待簽核],
    SUM(CASE WHEN fsu_status = '1' THEN 1 ELSE 0 END) AS [已簽核]
FROM edoc..flow_state_uid
WHERE fsu_did = @TargetFlowId
GROUP BY fsu_instord
ORDER BY fsu_instord ASC;

-- 【驗證 4】序號完整性 — 檢查是否有重複的 fsu_serial
SELECT 
    fsu_serial,
    COUNT(*) AS [重複數]
FROM edoc..flow_state_uid
WHERE fsu_did = @TargetFlowId
GROUP BY fsu_serial
HAVING COUNT(*) > 1;

-- 若此查詢有結果，表示存在重複序號，需清理
```

**驗證項目清單**：
- ✅ flow_instance.flow_instord 指向正確職級
- ✅ flow_state 應包含所需的所有職級（700、800、900 等）
- ✅ flow_state_uid 應有對應的簽核人員
- ✅ fsu_serial 應遞增無重複
- ✅ 無異常的缺漏職級或孤立記錄

## 🔑 核心概念回顧

### 三個重要表的角色

| 表名 | 用途 | 欄位例示 |
|------|------|-------|
| **flow_instance** | 申請單主檔 | flow_id, flow_instord（當前職級指標）, flow_createtime |
| **flow_state** | 關卡定義層 | fst_did, fst_instord（職級）, fst_isValid（條件是否進入） |
| **flow_state_uid** | 簽核人員層 | fsu_did, fsu_serial（流水號）, fsu_instord（職級）, fsu_status（0待簽/1已簽）|

### 關卡缺漏的常見原因

| 原因 | 症狀 | 解決方案 |
|------|------|--------|
| 判定腳本邏輯誤差 | fst_isValid = 0 | ➜ 修改判定腳本，重新送單 |
| 審批人員未派員 | fst_isValid = 1 但 flow_state_uid 無人 | ➜ 複製簽核紀錄或新增人員 |
| 前台顯示異常 | flow_instance.flow_instord 指向錯誤職級 | ➜ 同步更新主檔指標 |
| 序號衝突 | 新增時出現 UNIQUE KEY 錯誤 | ➜ 檢查 fsu_serial 是否重複 |
