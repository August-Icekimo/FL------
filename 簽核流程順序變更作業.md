# 📌 作業目的與情境

當前台 UI 呈現的簽核歷史、關卡順序因系統誤判或特殊維護需求需要手動調整（例如：上版作業檢查表 bfm_patchCheck），且在不破壞歷史流水號 (fsu_serial) 的前提下，必須透過資料庫直接調整關卡節點順序 (fsu_instord)，使前台能以正確的順序呈現簽核歷程。

# ⚠️ 核心限制與安全防禦

1-JOIN 限制：由於 Web 執行環境（如 Iron.aspx 介面）限制，關聯查詢最多僅能執行 單次 JOIN (1-JOIN Rule)。

唯一鍵衝突防範：在更新目前關卡節點順序時，切勿直接執行相互對調，否則會觸發 Violation of PRIMARY KEY / UNIQUE constraint 錯誤。本 SOP 採用安全、具緩衝區的 「三步緩衝換位法（Buffer Swap）」。

精準包夾條件：在執行 UPDATE 時，必須同時在 WHERE 條件中鎖定 did（流程 ID）、serial（流水號）與 uid（人員識別碼）三個維度，防止大範圍誤改。

## 🔍 第一階段：搜尋與定位（Locate & Inspect）

### 步驟一：模糊查詢並確認單檔資訊

企圖 (Intent)：透過使用者回報的前台單號，於表單主檔確認該單據確實存在，並抓出系統底層關鍵關聯識別碼 did。

操作步驟 (Steps)：

修改變數 @TargetCode 為目標單號。

執行查詢，確認單據主表 baseform..bfm_patchCheck 中對應的實體資料與核心文號 bfpc_did。
```SQL
-- ==========================================
-- 步驟一：模糊查詢表單主檔，取得關鍵對應 did
-- ==========================================
DECLARE @TargetCode NVARCHAR(50) = 'E20260080'; -- 🔍 請在此輸入前台申請單號

SELECT TOP 100 
    bfpc_id,
    bfpc_code AS [單號],
    bfpc_did AS [流程ID_did],
    bfpc_verify AS [核准狀態],
    flow_instord AS [當前主檔指標節點]
FROM baseform..bfm_patchCheck
WHERE bfpc_code LIKE @TargetCode + '%';
```

---

###步驟二：全方位探查流程狀態（符合 1-JOIN 限制）

企圖 (Intent)：在不違反 1-JOIN 限制下，交叉比對「主檔狀態」與「流程狀態紀錄表 flow_state_uid」，抓出需要對調順序的兩筆簽核歷史流水號（fsu_serial）與當前順序（fsu_instord）。

操作步驟 (Steps)：

將步驟一查得的 bfpc_did 帶入變數 @TargetDid。

執行查詢，觀察並記錄需要對調的 A、B 兩名人員其各自的 fsu_serial 與 fsu_uid。

```SQL
-- ==========================================
-- 步驟二：1-JOIN 綜合探查簽核歷程與節點全貌
-- ==========================================
DECLARE @TargetDid INT = 644897; -- 🔍 請輸入步驟一查得的 bfpc_did / fsu_did

SELECT 
    P.bfpc_code AS [單號],
    P.bfpc_did AS [主檔_did],
    P.bfpc_verify AS [主檔核准狀態],
    S.fsu_serial AS [簽核流水號],
    S.fsu_instord AS [關卡節點順序],
    S.fsu_uid AS [人員uid],
    S.fsu_status AS [簽核狀態]
FROM baseform..bfm_patchCheck P
INNER JOIN edoc..flow_state_uid S ON P.bfpc_did = S.fsu_did
WHERE S.fsu_did = @TargetDid
ORDER BY S.fsu_instord ASC;
```

## 🛠️ 第二階段：執行順序變更（Execute Swap）

### 步驟三：三步緩衝換位法（Buffer Swap）

企圖 (Intent)：在有 Unique Constraint 限制的情況下，安全對調兩個節點的順序。透過將第一位人員（A）先移入「未占用的緩衝節點（Buffer）」，騰出位置後將第二位人員（B）移入 A 的原位置，最後再將 A 從緩衝區移至 B 的原位置，達成安全對調。

操作步驟 (Steps)：

輸入並核對兩筆需要對調的流水號、UID 與目標關卡數值。

依序執行 Step 1 ~ Step 3 三個 UPDATE 段落。

注意：此異動操作被包裹在 Transaction 中，請於執行完畢確認筆數與結果正確無誤後，手動執行 COMMIT 遞交資料；如有異常則執行 ROLLBACK。

```SQL
-- ==========================================
-- 步驟三：三步緩衝換位法安全變更節點順序
-- ==========================================
BEGIN TRANSACTION;

DECLARE @TargetDid INT       = 644897;   -- ⚙️ 流程關聯 ID (fsu_did)
DECLARE @BufferOrd INT       = 504;      -- ⚙️ 緩衝/未占用的臨時關卡順序 (Buffer)

-- 節點 A 參數 (第一位要被調換的人員，例如原本在 502 關卡)
DECLARE @SerialA INT         = 4134417;  -- A 的簽核流水號 (fsu_serial)
DECLARE @UidA INT            = 942;      -- A 的人員 ID (fsu_uid)
DECLARE @TargetOrdA INT      = 503;      -- A 最終要換去的新關卡順序

-- 節點 B 參數 (第二位要被調換的人員，例如原本在 503 關卡)
DECLARE @SerialB INT         = 4137825;  -- B 的簽核流水號 (fsu_serial)
DECLARE @UidB INT            = 975;      -- B 的人員 ID (fsu_uid)
DECLARE @TargetOrdB INT      = 502;      -- B 最終要換去的新關卡順序

-- 【Step 1】騰出空間：將 A 移至暫存緩衝區
UPDATE edoc..flow_state_uid 
SET fsu_instord = @BufferOrd 
WHERE fsu_did = @TargetDid 
  AND fsu_serial = @SerialA 
  AND fsu_uid = @UidA;

-- 【Step 2】佔據位置：將 B 移至 A 的原位置
UPDATE edoc..flow_state_uid 
SET fsu_instord = @TargetOrdB 
WHERE fsu_did = @TargetDid 
  AND fsu_serial = @SerialB 
  AND fsu_uid = @UidB;

-- 【Step 3】完成對調：將 A 從暫存緩衝區移至 B 的原位置
UPDATE edoc..flow_state_uid 
SET fsu_instord = @TargetOrdA 
WHERE fsu_did = @TargetDid 
  AND fsu_serial = @SerialA 
  AND fsu_uid = @UidA;

-- 🚨 檢查更動結果 (請圈選以下 SELECT 語法檢查，確認無誤再 COMMIT)
/*
SELECT fsu_serial, fsu_instord, fsu_uid, fsu_name 
FROM edoc..flow_state_uid 
WHERE fsu_did = @TargetDid;

-- 確實驗收成功後，請單獨執行下方指令：
COMMIT TRANSACTION;

-- 若發現異常，請單獨執行下方指令還原：
ROLLBACK TRANSACTION;
*/
```
### 步驟四：主檔指標同步更新（防卡單與顯示不一致）

企圖 (Intent)：當前台 UI 會依賴主表欄位（例如：baseform..bfm_patchCheck 裡面的 flow_instord 指標）判斷表單當前關卡時，若發生前台顯示異常，需手動同步此欄位。

操作步驟 (Steps)：

將目標 flow_instord（如變更後的當前關卡 1100 或最新節點值）帶入變數 @NewInstOrd。

執行更新，使主表與工作流引擎狀態同步。

```SQL
-- ==========================================
-- 步驟四：同步更新主檔流程指標欄位
-- ==========================================
DECLARE @TargetDid INT           = 644897; -- ⚙️ 目標 did
DECLARE @NewInstOrd NVARCHAR(10) = '1100'; -- ⚙️ 調整後應指向的正確主檔關卡指標

UPDATE baseform..bfm_patchCheck
SET flow_instord = @NewInstOrd
WHERE bfpc_did = @TargetDid;

-- 驗證結果
SELECT bfpc_code, bfpc_did, flow_instord 
FROM baseform..bfm_patchCheck 
WHERE bfpc_did = @TargetDid;
```
